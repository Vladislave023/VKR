# VKR — прототип подсистемы валидации входных данных для электронной библиотеки

## О проекте

Данный проект представляет собой прототип пользовательской подсистемы информационной системы поддержки сотрудников Отдела электронной библиотеки Научной библиотеки ДВФУ.

Основная задача подсистемы — обеспечить корректный ввод метаданных при подаче выпускной квалификационной работы и предварительную валидацию загружаемого файла до этапа проверки сотрудником библиотеки.

Проект разработан в рамках выпускной квалификационной работы по теме:

**«Разработка прототипа информационной системы поддержки для сотрудников Отдела электронной библиотеки Научной библиотеки ДВФУ. Подсистема валидации входных данных»**

---

## Назначение системы

Подсистема предназначена для решения следующих проблем текущего процесса ввода данных:

- пользователь может ввести некорректное ФИО;
- год работы может быть введён в неверном формате;
- название работы может содержать запрещённые символы или некорректный регистр;
- в качестве файла может быть загружен не PDF;
- отсутствует жёсткий контроль обязательных полей;
- часть данных сотрудникам библиотеки приходится исправлять вручную.

Подсистема переносит основную проверку на этап ввода пользователем, тем самым снижая количество ошибок и облегчая работу сотрудников.

## Логика работы системы

### Сценарий подачи заявки

1. Пользователь входит в систему.
2. Открывает страницу подачи ВКР.
3. Заполняет обязательные поля.
4. Выбирает:

   * уровень образования;
   * институт/школу;
   * направление/специальность;
   * кафедру/департамент.
5. Загружает один PDF-файл.
6. Система выполняет клиентскую и серверную проверку.
7. Если данные корректны — заявка сохраняется.
8. Пользователь может открыть карточку заявки и посмотреть статус.

### Логика выбора учебных данных

Поля зависят друг от друга:

1. **Уровень образования**
2. **Институт/школа**
3. **Направление/специальность** — формируется по выбранным уровню и школе
4. **Кафедра/департамент** — формируется по выбранной школе

## Модели данных

### Справочники

* `Institute` — институт / школа
* `Department` — кафедра / департамент
* `EducationLevel` — уровень образования
* `DocumentType` — тип документа
* `Specialty` — направление / специальность
* `Program` — связка *школа + уровень образования + направление*

### Основная сущность

* `Submission` — заявка пользователя на загрузку ВКР

В заявке хранятся:

* пользователь;
* ФИО автора;
* ФИО руководителя;
* название работы;
* год;
* количество страниц;
* выбранные справочники;
* файл;
* исходное имя файла;
* расширение файла;
* размер файла;
* статус;
* комментарий проверяющего;
* дата создания и изменения.

---

## Статусы заявки

В системе используются следующие статусы:

* `draft` — черновик
* `submitted` — отправлено
* `needs_fix` — требует исправлений
* `accepted` — принято

При необходимости список можно расширить, например статусом:

* `rejected` — отклонено

## Как запустить проект

### 1. Клонировать репозиторий

```bash
git clone https://github.com/Vladislave023/VKR.git
cd VKR
```

### 2. Создать виртуальное окружение

#### Windows

```bash
python -m venv venv
```

#### Linux / macOS

```bash
python3 -m venv venv
```

### 3. Активировать виртуальное окружение

#### Windows

```bash
venv\Scripts\activate
```

#### Linux / macOS

```bash
source venv/bin/activate
```

После активации в терминале должно появиться `(venv)`.

### 4. Установить зависимости

```bash
pip install -r requirements.txt
```

### 5. Применить миграции базы данных

```bash
python manage.py makemigrations
python manage.py migrate
```

### 6. Создать администратора

```bash
python manage.py createsuperuser
```

Эта команда создаёт учётную запись для входа в административную панель Django.

### 7. Импортировать справочники

Если в проекте используются CSV-файлы со школами, направлениями, уровнями образования и кафедрами, нужно положить их в папку `data/`, а затем выполнить:

```bash
python manage.py import_refs
```

### 8. Запустить сервер разработки

```bash
python manage.py runserver
```

После запуска проект будет доступен по адресу:

```text
http://127.0.0.1:8000/
```

## Валидация

### Серверная валидация

На стороне Django выполняется полная проверка данных:

* корректность ФИО;
* корректность года;
* корректность количества страниц;
* корректность названия работы;
* допустимость выбранных учебных данных;
* принадлежность кафедры выбранной школе;
* принадлежность направления выбранной связке *уровень + школа*;
* проверка PDF-файла.

### Клиентская валидация

На стороне браузера выполняется:

* проверка при вводе;
* подсветка полей;
* вывод сообщений об ошибках без ожидания нажатия кнопки отправки;
* динамическая загрузка кафедр и направлений;
* вывод имени выбранного файла;
* очистка файла отдельной кнопкой.

---

## Проверка PDF-файла

В системе реализованы следующие ограничения:

* разрешён только **один файл**;
* разрешён только формат **PDF**;
* максимальный размер файла — **100 МБ**;
* файл проверяется по расширению;
* файл проверяется по MIME-типу;
* файл проверяется по сигнатуре `%PDF-`.

## Тестирование

При тестировании рекомендуется проверить:

### ФИО

* корректное полное ФИО;
* ввод латиницы;
* ввод инициалов;
* ввод капсом;
* ввод одиночных символов.

### Название работы

* название без кавычек;
* название в капсе;
* слишком короткое название;
* название с запрещёнными символами.

### Год

* корректный год;
* слишком маленький / слишком большой год;
* нечисловое значение.

### Файл

* загрузка PDF;
* загрузка не-PDF;
* файл больше 100 МБ;
* попытка отправки без файла.

### Каскадные списки

* выбор уровня образования;
* выбор школы;
* загрузка соответствующих направлений;
* загрузка соответствующих кафедр.
